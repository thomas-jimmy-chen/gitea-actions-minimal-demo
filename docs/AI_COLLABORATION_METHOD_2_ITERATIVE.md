# AI 協作工作流程方法 2：迭代式探索開發

**方法名稱**: Iterative Exploratory Development (IED)
**適用情境**: 需求不明確、需要邊做邊學時
**文檔版本**: 1.0
**建立日期**: 2025-12-31

---

## 1. 方法概述

當專案需求**尚未明確**或正在探索階段時，採用「小步快跑、頻繁迭代」的敏捷方法。透過與 AI 的持續對話，逐步釐清需求並實作。

---

## 3. 各階段詳細說明

### 3.1 Phase 1: 初步構想

**目標**: 建立對問題的初步理解

**AI 會問的問題**:
- 你想解決什麼問題？
- 有沒有類似的工具或系統可以參考？
- 你目前能想到的大概功能有哪些？
- 有沒有特別想先看到的效果？

**產出物**:
```markdown
## 初步理解

### 問題描述
用戶想要 [簡述問題]

### 初步想法
- 可能需要 A 功能
- 可能需要 B 功能
- 不確定的部分：...

### 第一個迭代目標
先做出 [最簡單的版本]，讓用戶看到效果
```

---

### 3.2 Phase 2: 最小實作

**目標**: 快速產出可運行的原型

**實作原則**:

| 原則 | 說明 |
|------|------|
| YAGNI | You Aren't Gonna Need It - 不預先實作未確認的功能 |
| 快速失敗 | 先做出來，有問題再改 |
| 可見結果 | 每次迭代都要有用戶可以看到/操作的東西 |
| 簡單優先 | 用最簡單的方法實現，不過度設計 |

**典型迭代範圍**:
- 1-3 個小時的工作量
- 1 個核心功能
- 可以運行並展示

---

### 3.3 Phase 3: 用戶反饋

**反饋類型與處理**:

```
反饋類型                          AI 的回應
─────────────────────────────────────────────────────
「這個可以，但我還想要...」   →  記錄需求，規劃下一迭代
「這不是我要的」             →  詢問差異，釐清需求
「我不確定」                 →  提供選項，讓用戶選擇
「很好，繼續」               →  進入下一個功能
```

**對話範例**:
```
用戶：這個按鈕的位置怪怪的
AI：  了解，你希望它放在哪裡？
      A) 右上角
      B) 頂部中間
      C) 其他位置（請描述）

用戶：B，而且希望顏色更明顯
AI：  好的，我來調整：
      - 位置移到頂部中間
      - 顏色改為藍色底白字
      你看這樣如何？[提供修改後的代碼]
```

---

### 3.4 Phase 4: 進入下一功能

**何時進入下一功能**:
- 當前功能用戶已滿意
- 或：當前功能可以暫時擱置
- 或：發現更重要的功能需要先做

**迭代結束檢查**:
- [ ] 當前功能可以運行
- [ ] 用戶理解目前的狀態
- [ ] 下一步方向已確認

## 5. 與 AI 協作的對話範本

### 開始專案

```
你：我想做一個 [模糊的描述]，但還不太確定具體要怎麼做

AI：好的，讓我們一步一步來。
    先告訴我：
    1. 你想解決什麼問題？
    2. 有沒有看過類似的工具？

    我們可以先做一個最簡單的版本，
    你看到效果後再告訴我要怎麼調整。
```

### 迭代中

```
AI：這是第一版 [功能描述]：
    [代碼或截圖]

    你覺得：
    A) 方向對了，繼續加功能
    B) 需要調整，[描述調整方向]
    C) 不是我要的，我想要的是...

你：B，我希望 [描述調整]

AI：了解，讓我修改...
    [修改後的版本]
```

### 遇到不確定

```
你：我不確定這裡要怎麼處理

AI：沒關係，我提供幾個選項：

    選項 A: [描述 + 優缺點]
    選項 B: [描述 + 優缺點]
    選項 C: [描述 + 優缺點]

    你傾向哪一個？
    或者我們可以先用最簡單的 A，
    之後有需要再改。
```

### 想要暫停

```
你：這個功能先這樣，我們來做另一個

AI：好的，目前狀態：
    - ✅ 功能 1（已完成）
    - 🔄 功能 2（進行中，可暫停）
    - ⏳ 功能 3（待開始）

    你想先做哪一個？
```

## 6. 迭代記錄範本

每次迭代建議記錄：

```markdown
## 迭代 #N - [日期]

### 本次目標
[這次迭代要做什麼]

### 完成內容
- [x] 功能 A
- [x] 調整 B
- [ ] 功能 C（移到下次）

### 用戶反饋
[用戶說了什麼]

### 下次方向
[下一迭代要做什麼]

### 開放問題
- 問題 1: ...
- 問題 2: ...
```

## 7. 何時轉換到結構化方法

當以下情況出現時，考慮轉換到方法 1（結構化規格驅動開發）：

- 需求已經大致明確
- 需要與他人協作
- 需要正式文檔
- 專案規模擴大

**轉換對話**:
```
AI：我注意到需求已經比較清楚了：
    - 功能 1: ...
    - 功能 2: ...
    - 功能 3: ...

    要不要整理成正式的規格文檔，
    這樣後續開發會更有條理？

你：好，請幫我整理

AI：[產出規格文檔，轉換到方法 1]
```

## 8. 常見陷阱與解決

| 陷阱 | 症狀 | 解決方法 |
|------|------|----------|
| 過度設計 | 還沒確認需求就寫複雜架構 | 回到最小實作原則 |
| 迭代太大 | 一次做太多功能 | 拆分，每次只做一件事 |
| 缺少反饋 | AI 自己做完全部 | 每個小步驟都確認 |
| 方向迷失 | 不知道現在在做什麼 | 停下來整理目前狀態 |
| 技術債累積 | 為了快而寫髒代碼 | 定期進行小幅重構 |

---

## 9. 與方法 1 的對比

| 面向 | 方法 1 (結構化) | 方法 2 (迭代式) |
|------|----------------|----------------|
| 需求明確度 | 高 | 低 |
| 前期文檔 | 完整規格 | 簡要記錄 |
| 開發節奏 | 規劃 → 實作 | 實作 → 反饋 → 調整 |
| 變更成本 | 高（需改規格） | 低（隨時可調） |
| 適合專案 | 企業專案 | 個人/探索專案 |
| AI 角色 | 規格產出者 | 對話夥伴 |

---

## 10. 小結

迭代式探索開發的核心精神：

```
        ┌─────────────────────────────────┐
        │                                 │
        │   「不用一開始就想清楚，          │
        │    先做出來看看，再慢慢調整」     │
        │                                 │
        └─────────────────────────────────┘
```

這個方法適合：
- 不確定自己要什麼的專案
- 需要邊做邊學的情境
- 快速驗證想法
- 與 AI 持續對話的開發模式
